# Dashboards e métricas — classes de bugs recorrentes

Bugs que já apareceram mais de uma vez, em telas diferentes, com a mesma causa raiz. Antes de construir uma métrica nova, checar esta lista.

## Multi-loja
- Código que só usa a **primeira** loja/`empId` selecionada em vez de iterar/agregar todas as selecionadas — apareceu em pelo menos 5 endpoints diferentes (vendas, comissão, financeiro, listagem de vendas, top-produtos). Corrigido em `/api/dashboard/vendas`, `/api/dashboard/top-produtos` e `/api/dashboard/charts` (2026-07-06) com merge/agregação em memória entre lojas — ver esses arquivos como referência de padrão.
- TOP N calculado **por loja antes de mesclar** esconde o líder real quando o cálculo deveria ser sobre o conjunto agregado. Sempre agregar entre lojas primeiro, TOP N depois.
- Três dashboards (Home, Vendas, Clientes) chegaram a usar três definições diferentes e silenciosamente divergentes de "recorrência" — definições de métrica cross-dashboard não convergem sozinhas, precisam ser reconciliadas explicitamente.
- **`COUNT(DISTINCT ...)` não pode ser somado entre lojas** — só é aditivo quando não há repetição entre os conjuntos. Um cliente que compra em 2 lojas selecionadas é contado 2x se cada loja roda `COUNT(DISTINCT clienteId)` separadamente e o resultado é somado em JS. Correção: cada loja devolve a **lista** de IDs distintos (não a contagem), e o chamador consolida um `Set`/`Map` único — chaveado por `${bridgeUrl}::${id}` (não só o ID), porque o mesmo ID numérico em bridges/bancos físicos diferentes não representa o mesmo cliente real. Corrigido em `/api/dashboard/kpis` e `/api/home/summary` (2026-07-06), incluindo a classificação de recorrência (que precisa da contagem all-time consolidada *antes* de classificar novo/recorrente, não classificar por loja e depois somar).
- Atribuir a "loja" de um evento histórico de um cliente (ex.: em que loja foi a 1ª compra) via `MIN(empId)` entre todas as vendas do cliente está **errado** — pega a menor `empId` que ele já usou algum dia, não a loja do evento em questão. Sempre fazer o JOIN de volta na venda real do evento (mesma data/condição) e pegar o `empId` **dessa** venda especificamente.
- Uma tabela de associação cliente↔loja (ex.: `cliente_empresa`) pode ter **uma data diferente por linha** para o mesmo cliente (uma por filial/empId) — agregar "novos cadastros" por mês sem colapsar para uma data por cliente primeiro conta o mesmo cliente como "novo" em vários meses ao mesmo tempo quando 2+ lojas estão selecionadas.

## Categorias que deveriam ser mutuamente exclusivas, mas não são
Um padrão de bug específico: um KPI tipo donut/pizza espera que N categorias somem 100% do total (ex.: saúde de estoque: abaixo/acima/sem mínimo/negativo/regular), mas cada categoria é escrita como uma condição SQL independente (`SUM(CASE WHEN ... THEN 1 ELSE 0 END)`) sem garantir que as condições sejam mutuamente exclusivas entre si. Resultado: a soma dos segmentos passa do total (ex.: quase 2x), e o gráfico/legenda mostra percentuais errados. Causa raiz típica: duas categorias usam condições que se sobrepõem em um caso de borda (ex.: "sem mínimo" e "regular" ambas verdadeiras quando o mínimo não está informado; "negativo" e "abaixo do mínimo" ambas verdadeiras quando o estoque é negativo E há mínimo informado). Correção: replicar a mesma cadeia de prioridade if/elif que uma implementação em JS correta usaria (checar a condição mais específica primeiro e excluir explicitamente das condições seguintes com `AND NOT (...)` ou faixas que não se tocam), em vez de condições independentes. Sempre validar somando os `SUM(CASE...)` de todas as categorias e comparando com `COUNT(*)` total antes de confiar no gráfico.

## Data / timezone
- Bug recorrente de rollover UTC-3: construir limites de dia com `toISOString()` no lado cliente ou servidor pode virar o dia (23:59:59 local vira o dia seguinte em UTC). Padrão de correção: usar getters locais (`getFullYear/getMonth/getDate`) no cliente, e construir `Date` ao meio-dia (`T12:00:00`) ao fazer parse de strings `YYYY-MM-DD`.
- Mesmo bug, variante servidor: `new Date().getFullYear()/getMonth()/getDate()` no servidor **não é seguro** — a Vercel roda em UTC por padrão (sem `TZ` configurada), então esses getters devolvem o calendário UTC, não o de Brasília. Entre 21h e meia-noite (horário de Brasília), "hoje" no servidor já virou o dia seguinte, adiantando filtros rápidos (Hoje/Este mês/Últimos N dias) em 1 dia. Correção validada: `new Intl.DateTimeFormat("en-CA", { timeZone: "America/Sao_Paulo" }).format(new Date())` devolve `YYYY-MM-DD` já ancorado no fuso certo, independente do fuso do processo — usar isso como base antes de qualquer aritmética de data no servidor (ver `lib/utils/format.ts::getDateRange`).
- "Período anterior" deve ser derivado das datas já resolvidas da requisição atual (`currentStart/currentEnd`), não de `new Date()` no servidor — usar "agora" do servidor é frágil quando o cliente já escolheu um range específico.

## Formatação / visual
- `Intl.NumberFormat('pt-BR', {style:'currency'})` insere um espaço non-breaking entre "R$" e o número — o navegador nunca encontra ponto de quebra e o valor vaza para fora do card em vez de quebrar linha. Causa raiz recorrente de "valor cortado" em KPIs.
- Nunca abreviar valores monetários em cards — sempre mostrar o valor completo e redimensionar o layout se precisar (preferência explícita do usuário, repetida em mais de uma sessão).
- CSS Grid com `align-items: stretch` (padrão) + `h-full` nos cards estica forçadamente cards curtos para igualar o mais alto da mesma linha — causa "cards desalinhados". Corrigir com `items-start` ou sistema de altura fixa explícito, não com cap de altura por componente.
- Recharts (`<Tooltip>` flutuante/absolutamente posicionado) quebra o layout do card ao passar o mouse em pizza/donut pequenos — usar `activeShape` (Sector que expande dentro do próprio SVG) em vez do Tooltip nativo para composições compactas.
- Recharts v3 não resolve `var()` de CSS custom properties dentro de gradientes SVG — cores de gráfico devem ser hex fixo por tema claro/escuro (convenção já usada em outros gráficos do projeto).
- Contraste claro/escuro: usar variáveis CSS de tema, não hex fixo — títulos/subtítulos hardcoded em branco ficam invisíveis no tema claro e vice-versa.

## Contagem / agregação SQL
- Join de `venda` com `vendaItem` para KPI por venda infla contagem (venda de 5 itens conta 5x). Separar granularidade de venda da granularidade de item.
- Range de data sem limite inferior causa timeout de full table scan — sempre ancorar em janela com índice.

## Processo de validação
Todos os itens acima foram descobertos **depois** de a feature já estar em produção (via print do usuário comparando com o ERP real, ou via log de erro da Vercel). O padrão que teria evitado a maioria: reconciliar o número calculado contra a tela real do MaxManager (ou contra uma query já validada, como "top 50 vendedores") **antes** de dar a feature como pronta — ver [erp-maxmanager-schema.md](erp-maxmanager-schema.md) e [collaboration-workflow.md](collaboration-workflow.md).
